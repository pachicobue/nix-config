---
name: jj-commit
description: >
  Jujutsu (jj) を使って現在の作業コピーの変更をコミットする。変更内容を分析して論理的なグループに分割し、
  各グループに適切なコミットメッセージを生成して `jj split` と `jj describe` で適用する。
  push は行わない（GPG署名が必要なため手動）。
  ユーザーが「コミットして」「jjでコミット」「jjで保存して」「jjで〜〜」のように指示したときに使う。
  git commit ではなく jj を使うよう指示された場合も必ずこのスキルを使うこと。
context: fork
---

## ワークフロー

### 1. 変更の確認

```bash
jj status
jj diff --stat
```

変更がなければ「変更はありません」と伝えて終了する。

### 2. コミットスタイルの確認

```bash
jj log -n 8 --no-graph -T 'description ++ "\n"'
```

リポジトリの既存コミットから言語・フォーマットを読み取る。このリポジトリでは通常 `scope: 日本語の説明` 形式。

### 3. 差分の詳細確認

```bash
jj diff
```

全変更内容を把握してグルーピングを計画する。

### 4. コミットの分割と記述

変更を論理的なグループに分けて適用する。

**グループが1つの場合（分割不要）:**
```bash
jj describe -m "メッセージ"
```

**グループが2つ以上の場合:**
```bash
# 最初のグループのファイルを切り出す（残りは @ に留まる）
jj split <group1-files...> -m "最初のコミットメッセージ"

# 次のグループを切り出す（さらに分割が必要な場合）
jj split <group2-files...> -m "2番目のコミットメッセージ"

# 最後に残った @ を記述
jj describe -m "最後のコミットメッセージ"
```

### 5. 確認

```bash
jj log -n 8
```

コミットが意図通りに作成されたか確認する。

## 分割の判断基準

**分割すべき場合:**
- 独立した機能・サービスへの変更が混在している（例：immich の設定変更と blueman の変更）
- 修正と新機能が混在している
- 異なるスコープの変更（例：`modules/` の変更と `hosts/` の変更が別の目的）

**分割しなくてよい場合:**
- 1つの機能追加に必要なファイル群がまとめて変更されている（モジュール + ホスト設定など）
- リファクタリングが複数ファイルにまたがっているが目的が1つ
- 関連する設定が一緒に変更されている

## コミットメッセージの書き方

- リポジトリの既存スタイルに合わせる（通常 `scope: 説明`）
- 「何を変えたか」より「何のための変更か」を簡潔に
- 日本語で書く（リポジトリの慣習に従う）

## 絶対にやらないこと

- `jj git push` / `git push` は実行しない（GPG署名が必要なため手動で行う）
- `--no-gpg-sign` などの署名スキップオプションは使わない
